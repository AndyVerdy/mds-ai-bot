# MDS AI Bot — Technical Documentation

## What It Is

MDS AI Bot is an AI-powered knowledge search engine for Million Dollar Sellers (MDS). It ingests transcripts from MDS mastermind sessions, expert calls, and presentations, then lets users ask natural-language questions and get answers sourced directly from that content.

**Live URL:** Hosted on Render (auto-deploys from GitHub)
**Repo:** https://github.com/AndyVerdy/mds-ai-bot

---

## Architecture Overview

The system uses a **RAG (Retrieval-Augmented Generation)** pipeline:

```
User Question
     │
     ▼
┌─────────────────┐
│  Flask Web App   │  (web.py)
│  or Widget API   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   ChromaDB       │  Similarity search using all-MiniLM-L6-v2
│   Vector Store   │  (free, runs locally — no API key needed)
└────────┬────────┘
         │  Top 5 most relevant chunks
         ▼
┌─────────────────┐
│  Claude Sonnet 4 │  Anthropic API generates answer
│  (LLM)           │  from retrieved context only
└────────┬────────┘
         │
         ▼
   Answer + Sources + Confidence Score
```

**Key principle:** The LLM never uses its own knowledge. It only answers based on the retrieved transcript chunks. If the content doesn't match the question, it says so.

## How It Works — Step by Step

### 1. Data Collection (Otter.ai Export)

Transcripts come from Otter.ai, which records MDS mastermind sessions. The export script (`otter_export.py`, not in git) connects to the Otter.ai internal API:

- **Authenticates** with email/password
- **Paginates** through all speeches using `last_load_ts` + `modified_after` cursor parameters
- **Filters out** L-10 and Advisory Council meetings
- **Saves** each transcript as a `.txt` file named `YYYY-MM-DD_Title.txt`
- **Skips** already-downloaded files (resume-safe)

**Result:** 331 transcript files covering January 2025 through March 2026.

### 2. Document Ingestion (`ingest.py`)

The ingestion pipeline converts raw files into searchable vector embeddings:

**Step 1 — Parse documents** based on file type:
- `.txt` files: Detects Otter.ai format (speaker + timestamp pattern), parses into segments
- `.vtt/.srt` files: Parses subtitle timestamps into segments
- `.pdf` files: Extracts text page by page (PyMuPDF)
- `.pptx` files: Extracts text slide by slide (python-pptx)
- `.md` files: Reads as plain text

**Step 2 — Extract metadata:**
- Speaker name extracted from filename (e.g., `Josh Hadley_otter_ai.txt` → "Josh Hadley")
- Date, event, and topic loaded from `metadata.json`
- A context header is prepended to each chunk: `[Speaker: Josh Hadley | Event: Mogul Call | Date: March 2025]`

**Step 3 — Chunk documents:**
- Transcripts: Already chunked by speaker/timestamp segments (~2,000 characters each)
- Other documents: Split using `RecursiveCharacterTextSplitter` (2,000 chars, 300 overlap)
- Overlap ensures context isn't lost at chunk boundaries

**Step 4 — Embed and store:**
- Each chunk is converted to a 384-dimensional vector using **all-MiniLM-L6-v2** (runs locally, no API)
- Vectors + text + metadata stored in **ChromaDB** (local file-based database)
- Processed in batches of 100

**Result:** 331 files → 9,879 searchable chunks.

### 3. Query Processing (`query.py`)

When a user asks a question:

**Step 1 — Similarity search (free, local):**
- The question is embedded using the same all-MiniLM-L6-v2 model
- ChromaDB finds the 5 most similar chunks using L2 distance
- No API call needed — this runs entirely locally

**Step 2 — Confidence check:**
- ChromaDB returns a distance score for each chunk (0 = perfect match, 2 = unrelated)
- Converted to confidence: `confidence = 1 - (distance / 2)`
- If average confidence < 0.15, returns "not enough information" without calling Claude

**Step 3 — Generate answer (Anthropic API call):**
- The 5 retrieved chunks are formatted as context with metadata
- Sent to Claude Sonnet 4 with a system prompt that says:
  - Only answer from the provided context
  - Reference speakers by name
  - Don't make up information
  - Don't list sources (the UI handles that)
- Claude generates a natural-language answer

**Step 4 — Build response:**
- Answer text from Claude
- Source list: deduplicated speakers with dates, events, topics
- Confidence score (0-1 scale)
- Number of chunks used

### 4. Web Interface (`web.py`)

Two ways to access the bot:

#### Full-Page Chat UI (`/`)
- Landing page with centered search bar and topic suggestions
- Topics auto-generated by Claude from the knowledge base (cached)
- Popular searches shown based on query tracking
- Transitions to chat mode after first search
- Displays answers with formatted markdown, source cards, and confidence bars
- Responsive design (mobile-friendly)
- Design: shadcn-style (white background, black text, zinc palette)

#### Embeddable Widget (`/widget.js`)
- Self-contained JavaScript file that creates a floating chat button
- Drop `<script src="https://your-app.onrender.com/widget.js"></script>` on any page
- Opens a chat panel in the bottom-right corner
- Same functionality as the full UI (questions, answers, sources, confidence)
- CORS enabled so it works on any domain

### 5. Deployment (Docker + Render)

**Build process:**
1. Docker starts from `python:3.12-slim`
2. Installs Python dependencies from `requirements.txt`
3. Copies all `.py` files and the `data/` directory
4. **Builds the vectorstore at image build time** — runs ingestion during `docker build`
5. This means the vectorstore is baked into the Docker image (no runtime ingestion needed)

**Runtime:**
- Gunicorn serves the Flask app on port 8000
- 1 worker, 120-second timeout (Claude can take time to respond)
- Render's free tier auto-deploys when code is pushed to GitHub `main` branch

**Important:** The vectorstore is NOT stored in git. It's rebuilt every time the Docker image is built. The raw transcript files in `data/` ARE in git because they're needed for this build-time ingestion.

## API Endpoints

### `POST /api/ask`
Ask a question to the knowledge base.

**Request:**
```json
{
  "question": "What did Ian Sells say about exit planning?"
}
```

**Response:**
```json
{
  "answer": "According to Ian Sells, exit planning should start...",
  "sources": [
    {
      "speaker": "Ian Sells",
      "date": "March 2025",
      "event": "Mogul Call",
      "topic": "",
      "type": "transcript",
      "source": "2025-03-15_Mogul Call with Ian Sells"
    }
  ],
  "confidence": 0.72,
  "chunks_used": 5
}
```

### `GET /api/suggestions`
Returns topic suggestions and popular searches for the landing page.

**Response:**
```json
{
  "topics": ["Exit Planning", "Amazon PPC", "Supply Chain Strategy", ...],
  "popular": ["how to scale", "amazon advertising", ...]
}
```

### `GET /api/health`
Health check endpoint.

**Response:**
```json
{
  "status": "ok"
}
```

### `GET /widget.js`
Returns the embeddable widget JavaScript. Add to any page:
```html
<script src="https://your-app.onrender.com/widget.js"></script>
```

## Configuration (`config.py`)

| Setting | Value | Description |
|---------|-------|-------------|
| `LLM_MODEL` | `claude-sonnet-4-20250514` | Claude model for answer generation |
| `LLM_TEMPERATURE` | `0.1` | Low temperature for factual answers |
| `CHUNK_SIZE` | `2000` | Characters per chunk (larger = more context) |
| `CHUNK_OVERLAP` | `300` | Character overlap between chunks |
| `TOP_K` | `5` | Number of chunks retrieved per question |
| `CONFIDENCE_THRESHOLD` | `0.15` | Minimum confidence to send to Claude |
| `COLLECTION_NAME` | `mds_knowledge_base` | ChromaDB collection name |

## CLI Commands (`bot.py`)

```bash
# Ingest documents into the knowledge base
python bot.py ingest data/otter-export-new/

# Ask a single question
python bot.py ask "What did Josh Hadley talk about?"

# Interactive chat mode
python bot.py chat

# Show knowledge base stats
python bot.py status

# Clear the knowledge base
python bot.py reset

# Start the web UI
python bot.py web 5001
```
